OVERVIEW

Project overview

YCS Enterprise Lab represents a small-to-medium organization planning to host infrastructure in Microsoft Azure. The organization needed two logically separated environments — Production and Development — deployed under a consistent governance model: standardized naming, mandatory tags, approved regions, protection against accidental deletion, controlled access, and proactive cost monitoring.

In scope

Resource groups, naming standards, tagging strategy, custom & built-in Azure Policy, location restrictions, tag inheritance, resource locks, budgets & cost alerts, RBAC, and compliance validation.

Out of scope

Virtual machines, Microsoft Entra hybrid identity, Azure Backup & Site Recovery, Azure Monitor, Defender for Cloud, Sentinel, application/database deployment, and production-grade connectivity.

Business requirements

  • Environment separation — Production and Development resources logically separated.
  • Standardized naming — every resource follows an identifiable naming convention.
  • Regional control — deployments restricted to Qatar Central or UAE North.
  • Tag governance — mandatory metadata: Environment, Department, Owner, Cost Center, Criticality, Project.
  • Environment validation — the Environment tag accepts only Production or Development.
  • Tag inheritance — child resources inherit the Environment tag from their parent resource group when missing.
  • Resource protection — Production resources protected against accidental deletion.
  • Cost control — monthly budget with cost notifications.
  • Access governance — administrative access follows RBAC and least-privilege principles.
  • Compliance visibility — Azure Policy compliance reviewed and documented.

SECTION 1

Planning & Design

The Planning and Design phase established the governance model, resource structure, naming standards, deployment boundaries, and logical architecture before any technical implementation began — separating Production and Development to improve resource organization, operational control, cost visibility, policy enforcement, security, and future scalability.

1.4 Subscription design

A single Azure subscription acts as the primary governance boundary for policy assignments, RBAC, cost management, budgets and alerts, resource inventory, and compliance reporting. Production and Development are separated through resource groups rather than separate subscriptions, since this is a controlled enterprise lab. In a larger production environment, separate subscriptions could instead be used for Production, Development, Security, Shared Services, Connectivity, and Disaster Recovery.

1.8 Tagging design

The Resource Group acts as the authoritative source for the Environment tag; Azure Policy inherits it automatically onto child resources that don't already carry it. Approved Environment values are restricted to Production or Development — preventing inconsistent values like "Prod", "DEV", "Testing", or "Production01".

Allow only Production and Development Environment values, preventing invalid classification.

03

Allowed Locations

Restricts deployments to Qatar Central and UAE North only.

04

Resource Lock

05

Budget & cost alerts

Monthly budget with notifications at 50%, 80%, and 100% thresholds.

06

RBAC

Owner, Contributor, and Reader roles applied under least-privilege.

Design decision — a single subscription with resource-group-level separation kept the lab controlled and easy to reason about, while every control (naming, tags, policy, locks, cost, RBAC) was still designed to scale cleanly into a multi-subscription enterprise landing zone later.

SECTION 2

Implementation

Step-by-step configuration of the Azure environment in the Azure Portal, following the design established in Section 1 — from initial resource groups through networking, tagging, policy, locks, cost controls, and access management.

2.1 Azure Portal Environment

The Azure Portal served as the primary interface for provisioning and managing all governance controls throughout this project.

2.2 Resource Group Provisioning

Two resource groups were provisioned to separate Production and Development workloads, in line with the environment design defined in Section 1.7.

2.3 Virtual Network & Subnet Configuration

Non-overlapping virtual networks were deployed for each environment, matching the address plan defined in Section 1.7.

2.4 Network Security Groups

A dedicated Network Security Group was associated with each environment to control inbound and outbound traffic at the subnet level.

2.5 Public IP Addresses

A static Public IP address was provisioned per environment to support future connectivity requirements.

2.6 Storage Accounts

A Storage Account was deployed in each resource group. Naming follows the exception defined in Section 1.6, as Azure Storage Account names cannot include hyphens.

2.7 Tagging Implementation

The approved tagging schema from Section 1.8 was applied directly to the resource groups, which act as the authoritative source for inherited metadata.

2.8 Azure Policy -- Definitions & Assignments

Custom and built-in Azure Policy definitions were created and assigned at the subscription scope to enforce the governance standards defined in Section 1.9.

2.9 Resource Locks

A CanNotDelete lock was applied to the Production resource group to protect critical resources from accidental deletion, as required in Section 1.9.

2.10 Cost Management & Budgets

A monthly budget with tiered alert thresholds was configured at the subscription scope to support the cost control requirement in Section 1.2.

2.11 Role-Based Access Control (RBAC)

Least-privilege access was enforced using built-in Azure roles, assigned directly on the resource groups as defined in Section 1.9.

SECTION 3

Governance Validation & Compliance

Once deployed, each governance control was validated to confirm it was enforced correctly — including negative test cases that prove non-compliant actions are blocked.

3.1 Policy Compliance Dashboard

Following policy assignment, the Azure Policy Compliance dashboard was used to validate that deployed resources conformed to the governance standards.

3.2 Tag Inheritance Validation

The tag inheritance policy was tested to confirm that child resources automatically receive the Environment tag from their parent resource group.

3.3 Cost Monitoring Validation

Budget alerts were validated to confirm that spending notifications trigger correctly at each configured threshold.

3.4 Resource Inventory & Cost Allocation

The final resource inventory was reviewed to confirm alignment with the intended architecture, and cost analysis was validated by resource group and tag.

3.5 Governance Control Testing

Each governance control was deliberately tested against a non-compliant scenario to confirm enforcement works as designed.

3.6 Environment Health Check

A final health check confirmed the Azure environment was stable throughout the governance validation process.

SECTION 4

Project Summary

This project delivered a complete Azure Governance foundation for the YCS Enterprise Lab — covering subscription-level design, environment separation, naming and tagging standards, policy enforcement, resource protection, cost governance, and role-based access control.

4.1 Outcomes achieved

Production and Development fully separated using dedicated resource groups, networks, and non-overlapping address spaces.

A consistent, predictable naming convention applied across every resource type.

Standardized tagging enforced through Azure Policy, including automatic tag inheritance.

Custom and built-in policies restricted deployments to approved regions and enforced valid Environment values — validated through negative testing.

A CanNotDelete lock protected the Production resource group from accidental deletion.

A monthly budget with 50/80/100% alert thresholds provided proactive cost visibility.

RBAC applied under least-privilege, with Owner, Contributor, and Reader roles assigned appropriately.

The Azure Policy Compliance dashboard continuously validated governance posture.
